## Agent Specifications

### `godot-animator`

**Distinguishing behavior.** Picks **AnimationPlayer vs AnimationTree** by complexity (single-clip vs blended graph), designs animation state machines and blend trees as data, reaches for the **IKModifier3D family** (CCDIK3D / FABRIK3D / JacobianIK3D â€” Godot 4.6+) when procedural targeting beats keyframing, uses **BoneConstraint3D** for copy/aim transforms, and is the agent for retargeting mocap to a rig. Distinguishes **animation FSM (inside AnimationTree)** from **gameplay FSM (state-machine skill)** and explicitly hands gameplay logic back to game-dev.

**Skills loaded:**

- Primary: `animation-system`, `tween-animation`
- Subsystem: `2d-essentials` (sprite animation), `3d-essentials` (skeletal context)
- Cross: `state-machine` (boundary), `godot-debugging`

**Trigger examples (3, matching existing format):**

- Designing a 3D character blend tree (locomotion + combat layers)
- Adding IK foot placement on uneven terrain (Godot 4.6+ FABRIK3D)
- Sprite animation with 8-direction blending for a top-down game

### `godot-csharp-engineer`

**Two declared modes.**

1. **User-code mode (default).** C#-first implementation. Writes `[Signal]` delegates (not GDScript translations), minimizes Variant marshalling, avoids per-frame allocations (GC pressure), uses partial classes for editor exports, prefers `async`/`Task` with `ToSignal` over GDScript-style coroutines.
2. **Parity mode.** Invoked when the user says "close C# parity for [skill]" or similar. Reads `docs/superpowers/notes/2026-04-30-csharp-parity-debt.md`, picks the requested section, writes the C# block per the per-row guidance there, runs `node scripts/validate-skills.mjs`, then updates the parity notes file to strike out the closed section.

**Skills loaded:**

- Primary: `csharp-godot`, `csharp-signals`
- Subsystem: whichever applies for the section/task
- Cross: `godot-optimization` (GC/perf), `godot-code-review`

**Trigger examples:**

- "Convert this GDScript player controller to idiomatic C#"
- "Why is my C# game allocating 50KB/frame?" (GC diagnosis)
- "Close C# parity for `skills/save-load/SKILL.md`" (parity mode)

### `godot-ui-designer`

**Distinguishing behavior.** **Always Control nodes** (never Node2D for UI), **container-driven layout** (no manual positioning), **theme resources** for centralized styling, anchor-preset / stretch-mode thinking for responsiveness, distinguishes **HUD (gameplay-coupled)** from **menu UI (scene-coupled)**, and hooks `TranslationServer` / RTL into UI from the start instead of retrofitting. Knows Godot 4.5 additions: **FoldableContainer**, **Stacked Label Effects**.

**Skills loaded:**

- Primary: `godot-ui`, `responsive-ui`, `hud-system`
- Cross: `localization` (TranslationServer, RTL), `tween-animation` (UI motion)

**Trigger examples:**

- Building a settings menu with FoldableContainer accordions (Godot 4.5+)
- Designing a localized HUD that handles RTL languages and 1.5Ă— plurals
- Responsive UI for a game shipping on desktop + mobile + Steam Deck

## Validation Strategy

Agents have no unit tests (they are prompts), so validation is layered:

- **Static.** `node scripts/validate-skills.mjs` after every commit. Frontmatter, structure, cross-references must pass.
- **Frontmatter-by-eye.** Each new agent ends with `model: inherit` and contains three `<example>` blocks with `<commentary>` (matching shader-author / performance-profiler template).
- **Smoke test (manual).** Before tagging, run each new agent with one of its trigger examples in a real Godot project. Confirm it loads the expected skills. The plan agent spells out which prompts.
- **Parity mode test.** Run `godot-csharp-engineer` in parity mode against one of the 10 sections. Confirm it (a) reads the notes file, (b) writes valid C#, (c) updates the notes file with a strikethrough.
- **Validator delta.** Before commit 6, baseline = 42 warnings. After = expected ~32. If it drops more or less than expected, investigate.

## Edge Cases and Risks

- **Parity mode discoverability.** If `godot-csharp-engineer`'s parity mode is buried in the system prompt, the harness may never invoke it. Mitigation: an explicit trigger phrase (`"close C# parity for ..."`) is called out in the description's `<example>` block, plus a CONTRIBUTING.md note that contributors closing parity should invoke this agent.
- **Routing collision.** With 8 agents, harness selection accuracy may degrade â€” overlapping descriptions cause flip-flopping between game-dev and a specialist mid-task. Mitigation: each new specialist's first description sentence starts with the distinguishing trigger ("Use this agent when..."), and the routing tweaks in commit 4 explicitly defer.
- **Release workflow secret.** If `MARKETPLACE_TOKEN` is unset, the workflow logs a warning and continues. Pre-flight: confirm secret status before tagging; fall back to manual marketplace bumps if absent.

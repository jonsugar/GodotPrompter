# v1.9.0 Design — "Native Power & Mobile Shipping"

**Status:** Approved (brainstorming complete 2026-05-29)
**Author:** Jan Mesarč
**Predecessor:** v1.8.0 (C# parity initiative complete + Codex sub-agents)

## Summary

A focused feature release that adds **3 new skills** in the *native power + platform shipping*
space, corrects the doc-accuracy issues left behind in the v1.8.0 release, and wires the new
skills into the existing cross-reference graph. The repo grows from **45 → 48 skills**. Godot
4.3+ baseline is unchanged. The validator target is unchanged: **0 errors, 10 warnings**
(the 10 are the intentional `csharp-parity-accepted` sections in `gdscript-patterns` and
`gdscript-advanced`); the new skills must not introduce new warnings.

Both multi-release initiatives that defined v1.6.0–v1.8.0 — C# parity and the token-budget /
Pattern X restructure — are complete, so v1.9.0 is a fresh feature direction rather than a
continuation.

## Goals

1. Fill the three clearest structural gaps in the skill catalog: native extensions,
   concurrency, and mobile platform shipping.
2. Correct the factual discrepancies introduced into the v1.8.0 docs (the planner predicted 5
   token-budget warnings that never materialized — the files landed just under budget).
3. Keep the cross-reference graph coherent: new skills link in, and the existing skills/agents
   that should mention them do.

## Non-goals (YAGNI)

- `platform-services` / Steam (GodotSteam) — deferred; pulls in third-party addons and is its
  own release-sized topic.
- A dedicated web / HTML5 export skill — deferred.
- A new dedicated agent (e.g. `godot-native-engineer`) — deferred; this is a focused release
  and the user did not pick the "new agents" theme.
- A deep Rust `gdext` tutorial — kept to a single `references/` section.
- Console / NDA platforms.

## The three new skills

Each follows the repository's standard structure: YAML frontmatter (`name` matching the folder),
a `**Related skills:**` line between the H1 and the first section, numbered sections with
GDScript-then-C# examples, an implementation checklist near the end, and a `SKILL.md` kept
**≤ 16 KB** (overflow split into `references/<topic>.md` per Pattern X).

### Skill 1 — `gdextension`

**Frontmatter description:** `Use when building native extensions for Godot — godot-cpp (C++) or gdext (Rust), binding classes, building, and GDScript/C# interop`

**Sections (core SKILL.md):**
1. **When to reach for GDExtension** — decision guidance vs GDScript / C# / engine modules
   (performance-critical native code, existing C/C++ libraries, hot loops). Cross-ref
   `gdscript-advanced`, `csharp-godot`, `godot-optimization`.
2. **Project setup** — `godot-cpp` as a git submodule, `SConstruct`, the `.gdextension` file
   layout, library naming per platform.
3. **Binding a class** — `GDCLASS`, `ClassDB` registration of properties / methods / signals
   (C++ example).
4. **Building & distributing** — multi-platform build matrix, `.gdextension` `entry_symbol`
   and per-platform library paths, debug vs release, ABI/version compatibility notes for
   Godot 4.x.
5. **Interop** — calling the extension from **GDScript** *and* **C#** (this is the section that
   carries the parity-paired blocks).

**References (load-on-demand):**
- `references/rust-gdext.md` — the Rust (`gdext`) alternative: `#[derive(GodotClass)]`, `#[func]`,
  `#[signal]`, Cargo setup.
- `references/debugging-native.md` — attaching a native debugger, sanitizer builds, common crash
  signatures.

**C# parity handling:** The primary language is C++ (with a Rust alternative). The validator's
parity rule only fires on a section that contains a `gdscript` fenced block but no `csharp`
block. Only the **Interop** section contains a `gdscript` block, and it gets a paired `csharp`
block. The C++/Rust sections contain no `gdscript` block, so they raise **no** parity warning —
**no allowlist entry is required.**

**Godot version note:** GDExtension is 4.1+; examples target 4.3+. Call out ABI compatibility
across 4.x point releases.

### Skill 2 — `multithreading`

**Frontmatter description:** `Use when running work off the main thread — WorkerThreadPool, Thread/Mutex/Semaphore, call_deferred, thread-safe scene access, and threaded resource loading`

**Sections (core SKILL.md):**
1. **Threading model & rules** — the main thread owns the scene tree; what is safe vs unsafe to
   touch from a worker; the golden rule (mutate the tree only on the main thread).
2. **WorkerThreadPool** (preferred high-level API) — `add_task`, `add_group_task`,
   `wait_for_task_completion`, parallel job patterns.
3. **Low-level `Thread` + `Mutex` + `Semaphore`** — when you need a long-lived dedicated thread;
   producer/consumer.
4. **`call_deferred` & safe hand-back** — getting results back onto the main thread safely.
5. **Threaded resource loading** — `ResourceLoader.load_threaded_request` /
   `load_threaded_get_status` / `load_threaded_get`, progress reporting.
6. **C# concurrency interop** — `System.Threading.Tasks` / async-await vs Godot `Thread`,
   `CallDeferred`, when to prefer .NET tasks.

**References (load-on-demand):**
- `references/pitfalls.md` — deadlocks, data races, `await` vs threads confusion, common Godot
  threading mistakes.

**C# parity handling:** Full GDScript + C# parity per section (C# has both `System.Threading`
and Godot's `CallDeferred`). No allowlist needed.

**Cross-refs:** `godot-optimization`, `assets-pipeline` (threaded loading), `csharp-godot`,
`gdscript-advanced` (async pitfalls).

### Skill 3 — `mobile-development`

**Frontmatter description:** `Use when targeting Android/iOS — export and signing, permissions, plugins, in-app purchases, ads, app lifecycle, device features, and mobile performance`

Scoped as **deltas from `export-pipeline`** (which already covers export presets and CI/CD), not
a duplicate.

**Sections (core SKILL.md):**
1. **Export & signing** — Android keystore/debug-keys, iOS provisioning/signing, the deltas
   beyond the generic `export-pipeline` flow.
2. **App lifecycle** — `NOTIFICATION_APPLICATION_PAUSED` / `RESUMED` / `FOCUS_IN` / `FOCUS_OUT`,
   autosave-on-background, releasing resources.
3. **Runtime permissions** — requesting Android permissions at runtime, checking grants.
4. **Plugins** — Android (AAR / Godot Android plugin) and iOS plugin overview, how Godot-side
   glue calls them.
5. **Monetization** — IAP and ads via plugins, presented as a **service-abstraction pattern**
   (keep game code platform-agnostic), not a full SDK walk-through.
6. **Device features** — vibration, accelerometer/gyro sensors, safe-area / notch handling via
   `DisplayServer`.
7. **Mobile performance budget** — Mobile/Compatibility renderer choice, texture compression,
   draw-call limits → cross-ref `godot-optimization`.

**References (load-on-demand):**
- `references/plugins.md` — authoring/integrating Android and iOS plugins in more depth.
- `references/iap-and-ads.md` — fuller IAP/ads integration flows behind the abstraction.

**C# parity handling:** Godot-side glue in GDScript + C#. Native plugin code (Kotlin/Java/Swift)
lives in language blocks that do not trigger the GDScript→C# parity check.

**Cross-refs:** `export-pipeline`, `responsive-ui` (safe areas), `input-handling` (touch),
`godot-optimization`, `assets-pipeline` (texture compression).

## Doc-accuracy fixes (correcting v1.8.0)

The v1.8.0 implementation plan predicted 5 skills would carry `token-budget-exceeded` warnings;
post-review trims (notably the player-controller Pattern X split) landed all 5 at 15.97–16.36 KB,
just under the 16384-byte limit. The prose was never corrected, so the shipped docs overstate the
warning count. The tag `v1.8.0` points at the final commit (`9b37843`); its actual validator
output is `0 errors, 10 warnings, 0 token-budget`.

**Decision (user-approved):** edit the v1.8.0 CHANGELOG entry **in place** to correct the numbers
(rather than leaving the wrong text and only noting a correction under v1.9.0).

- **`CHANGELOG.md`** v1.8.0 release note (line ~31):
  - `0 errors, 15 warnings (10 csharp-parity-accepted ... + 5 token-budget-exceeded pre-existing)`
    → `0 errors, 10 warnings (10 csharp-parity-accepted intentional GDScript-only; 0 token-budget)`.
  - `Down from 22 in v1.7.3` → `Down from 17 in v1.7.3` (22 was the v1.7.2 count).
  - Reword the closing sentence: "5 skills … carry pre-existing `token-budget-exceeded` warnings at
    16.1–16.3 KB" → "5 skills (`3d-essentials`, `ai-navigation`, `animation-system`,
    `multiplayer-basics`, `physics-system`) sit just under the 16 KB budget (15.97–16.36 KB) and
    remain Pattern X candidates for a future release."
- **`docs/superpowers/notes/2026-04-30-csharp-parity-debt.md`** (lines ~40 and ~42):
  - Apply the same `15 → 10`, drop-the-5-token-budget correction.
  - Name **both** `gdscript-patterns` *and* `gdscript-advanced` as the accepted GDScript-only
    category (line 42 currently names only `gdscript-patterns`).
- **`README.md`**:
  - Skills badge `45 → 48`; "**45 skills**" prose → "**48 skills**".
  - Add `gdscript-advanced` to the **Scripting** table (currently missing; 3 → 4 rows).
  - Add a new **Native & Performance (2 skills)** category: `gdextension`, `multithreading`.
  - Add `mobile-development` to **Build & Deploy** (4 → 5 rows).
  - New category-count audit must total 48: Core 6 + Architecture 6 + Physics/2D/3D/XR 4 +
    Gameplay 12 + UI 3 + Multiplayer 3 + Rendering 2 + Build&Deploy 5 + Scripting 4 + Math 1 +
    Native&Performance 2 = **48**.
  - Validation section baseline: `Current baseline (v1.7.3): 0 errors, 17 warnings` →
    `Current baseline (v1.9.0): 0 errors, 10 warnings (all csharp-parity-accepted)`.
  - Add a v1.9.0 line to the headline feature paragraph (native extensions + concurrency +
    mobile shipping).

## Validation & release mechanics

1. Each new `SKILL.md` ≤ 16 KB; overflow → `references/`.
2. Every `gdscript` block in a new skill has a paired `csharp` block (or the section uses a
   non-GDScript primary language and contains no `gdscript` block).
3. `node scripts/validate-skills.mjs` → **`Skills validated: 48`, `0 errors, 10 warnings`**
   (no new warnings beyond the existing 10 accepted).
4. `node scripts/count-tokens.mjs --tokenizer --markdown` → regenerate `docs/token-budget.md`;
   confirm 0 skills over budget.
5. `node scripts/bump-version.mjs 1.9.0` → bump `package.json`, `.claude-plugin/plugin.json`,
   `.claude-plugin/marketplace.json`, `.cursor-plugin/`, `.opencode/`, plus sibling marketplaces
   (skillsmith primary, godot-prompter-marketplace legacy).
6. New `CHANGELOG.md` `## [1.9.0]` entry.
7. Tag `v1.9.0` and push **only after explicit user confirmation** (push triggers
   `.github/workflows/release.yml`, which validates, creates the GitHub release, and opens
   marketplace PRs when `MARKETPLACE_TOKEN` is configured).
